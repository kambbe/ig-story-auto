# ig-story-auto — 営業案内ストーリーの自動投稿

その日の営業状況（営業中／臨時休業）を毎朝 Instagram ストーリーズに自動投稿する仕組み。
写真の上に、日付・OPEN/CLOSED・営業時間・次の営業日・一言（備考）を自動で載せて画像化する。

- **営業日** … `OPEN` ＋ 日付・営業時間
- **休業日** … `CLOSED` ＋ 次の営業日と営業時間
- 臨時休業・時間変更は **Googleスプレッドシート**に1行足すだけ（スマホから、GitHub不要）
- 背景写真は `backgrounds/` に入れておくと日替わりで自動差し替え（HEIC可）
- 毎朝 **JST 10:00** に GitHub Actions が自動投稿。トークンも毎月自動更新

日々の使い方・初期セットアップは **[運用ガイド.md](./運用ガイド.md)** を参照。

---

## 仕組み（データの流れ）

```
schedule.json（週の固定営業時間）
        ＋
Googleスプレッドシート（臨時休業・時間変更・備考）  ← スマホから編集
        ↓  src/render.mjs が毎朝読み込み
その日の状態を計算（src/status.mjs）
        ↓  src/template.html に流し込み
Playwright で 1080×1920 PNG を生成（背景は backgrounds/ から日替わり）
        ↓  src/post.mjs
Instagram グラフAPI（graph.instagram.com）でストーリー投稿
```

## ファイル

| ファイル | 役割 |
|----------|------|
| `schedule.json` | 店名・週の固定営業時間・スプレッドシートURL・（予備の）臨時設定 |
| `src/status.mjs` | 指定日の営業/休業と次の営業日を計算（JST基準） |
| `src/render.mjs` | スプレッドシート取得＋テンプレ流し込み＋PNG生成 |
| `src/template.html` | ストーリーの見た目（`:root` の変数で色・フォント・文字サイズを調整） |
| `src/post.mjs` | Instagram へストーリー投稿（2ステップ） |
| `src/refresh-token.mjs` | 長期トークンを延長（月次で自動実行） |
| `backgrounds/` | 背景写真（jpg/png/webp/heic、日替わりで1枚選択） |
| `.github/workflows/daily-story.yml` | 毎朝 JST 10:00 に投稿 |
| `.github/workflows/refresh-token.yml` | 毎月1日にトークンを自動更新 |

## ローカルで見た目を確認

```bash
npm install
npx playwright install chromium

node src/render.mjs              # 今日の分 → out/story.png
node src/render.mjs 2026-07-22   # 指定日でテスト → out/story-2026-07-22.png
```

`out/story-*.html` も出力されるので、ブラウザで開けば調整しやすい（原寸なのでズームアウト推奨）。
見た目は `src/template.html` 冒頭の `:root`（`--accent` / `--muted` / 各 `--fs-*`）で変える。

## スケジュールの持ち方

- **週の固定営業時間** … `schedule.json` の `regular`（`null` = 定休日）。
- **臨時休業・時間変更・備考** … スプレッドシート（`overridesSheetCsvUrl` に公開CSVのURL）。
  同じ日付はスプレッドシートが `schedule.json` の `overrides` より優先。取得失敗時は `overrides` で継続。

スプレッドシートの列は3つ：`日付 / 営業時間 / 備考`
- 臨時休業 = 日付だけ入れて営業時間は空欄
- 時間変更 = `14:00-18:00`（`〜` でも可）
- 備考 = 任意の一言（ストーリーに表示）

## 背景写真（`backgrounds/`）

`backgrounds/` に入れた写真が、ストーリーの背景として**日替わり**で使われる。
写真が1枚も無い場合は、黒基調（`theme.background`）で出力される。

**対応形式**：`.jpg` / `.jpeg` / `.png` / `.webp` / `.heic`（iPhoneのHEICは内部で自動的にJPEGへ変換）

### 選ばれ方（ランダムではない）

- ファイル名順に1列に並べ、**1日ごとに1枚ずつ順番に**使っていく。最後まで行ったら先頭に戻る。
- なので **10枚入れれば10日で一巡**し、その間に**すべての写真がちょうど1回ずつ**登場する。
  使われないまま眠る写真も、特定の写真だけ何度も出ることもない。
- 日付から決まる仕組みなので、**同じ日なら何度実行しても同じ写真**（当日に再実行しても変わらない）。

#### 「ファイル名順」とは

パソコンの文字コード順に並ぶ。ざっくり **数字 → 英大文字 → 英小文字 → 日本語** の順。

```
01_店構え.jpg → 02_作業風景.jpg → 1.jpg → 10.jpg → 2.jpg
→ Banana.jpg → IMG_10.HEIC → IMG_2.HEIC → apple.jpg → あさ.jpg → 春.jpg
```

- 数字は「文字」として比べるので、**`10.jpg` は `2.jpg` より前**に来る（`IMG_10` が `IMG_2` より前になるのも同じ理由）。
- 英字は**大文字が先**（`Banana` → `apple`）。日本語のファイル名は**いちばん後ろ**にまとまる。
- **順番をきっちり決めたい**なら、`01_`, `02_`, `03_` … のように**桁を揃えた番号**を先頭に付けるのが確実。

ただし、どんな順番でも**一巡すれば全部が1回ずつ**使われる。並び順は「どの日にどれが出るか」を決めるだけなので、こだわりがなければ気にしなくてよい。

### 追加・削除・差し替えしたとき

「どれを使ったか」は記録していない（毎回その日だけ計算する）ので、フォルダをいじると次のようになる。

| 操作 | どうなるか |
|------|-----------|
| **写真を追加** | 翌日から巡回に加わる。枚数が変わるので「どの日にどの写真か」の対応はズレるが、**追加分も含めて全部使われる**。 |
| **写真を削除** | 残った写真で巡回を続ける。消した写真が出なくなるだけで、問題は起きない。 |
| **同じ名前で差し替え** | その枠がそのまま新しい写真になる。 |
| **名前を変更** | ファイル名順が変わるので、並び順（＝どの日に出るか）がズレる。 |

注意点は1つだけ。枚数が変わると割り当てが計算し直しになるため、**追加・削除の直後は、最近出た写真がいつもより少し早く再登場することがある**。壊れたり、永久に使われない写真が出たりはしない。

### 写真選びのコツ

- 縦向き（9:16）を意識。**中央は文字が乗るので、すっきり空いている構図**が合う。
- 明るすぎない写真の方が文字が映える（上に暗いグラデも自動でかかる）。
- 年中使える定番（店構え・商品・作業風景）を数枚入れておくと、入れ替えをサボっても成立する。

## Instagram 側（Instagramログイン方式）

- Instagram プロアカウント＋Facebookページ連携。
- Meta アプリ（ユースケース「Instagramでメッセージとコンテンツを管理」）。
- 自分のアカウントに投稿するだけなら **App Review・ビジネス認証は不要**（開発モードのまま）。
- 必要な GitHub Secrets：`IG_USER_ID` / `IG_ACCESS_TOKEN` / `GH_PAT`（トークン自動更新用）。
- トークンは約60日で失効するが、`refresh-token.yml` が毎月更新するので基本は放置。

詳しい取得・設定手順は **[運用ガイド.md](./運用ガイド.md)** を参照。

## 休業日の投稿

`schedule.json` の `postClosedDays` で切り替える。

**既定（`false`）の挙動 ＝ 賢い分岐：**
- **定休日**（`regular` の休み）… 投稿しない（常連は分かっているので、投稿がない＝休み）
- **臨時休業**（シート/overrides で入れた休み）… `CLOSED` ＋次の営業日を**投稿する**（イレギュラーは告知したいから）
- 営業日 … `OPEN` を投稿

`true` にすると、定休日も含めてすべての休業日を毎日投稿する。

判定は `status.mjs` の `temporaryClosure`（overrides由来の休業か）で行い、
定休日スキップ時は `render.mjs` が画像を作らず、ワークフローが投稿ステップをスキップする。

## 失敗通知（推奨・自分で設定）

放置運用なので、投稿が失敗したとき（トークン失効、シートURL変更など）に気づけるよう
GitHub の失敗通知をオンにしておくと安心。

1. GitHub 右上アイコン → **Settings**（アカウント設定）→ **Notifications**。
2. **Actions** の項目で「**Send notifications for failed workflows only**」にチェック（メール等で受け取る）。

これで、毎朝のワークフローがコケた朝だけ通知が届く。成功時は静かなまま。

## メモ

- ストーリーへのリンクスタンプはAPIでは付けられない（自動投稿では不可）。オンライン導線はプロフィールのリンクへ。
- 定刻実行は数分ずれることがある。当日ぶんの変更はスプレッドシートに早めに。
